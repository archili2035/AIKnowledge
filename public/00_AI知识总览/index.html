当前原型定位

这是一张总览地图，不是最终知识正文。重点是回答“AI 版图大概怎么分”，而不是在首页塞进所有细节。

建议阅读路径：先在左侧看版图，再根据兴趣钻到专题页或节点页。

双层方案：底层可维护，上层可总览

你之前想要的 XMind 感，其实更适合放在展示层；而长期维护需要的结构化、可扩展、可回查，则应该放到底层数据层。

底层存储层

原子条目 + 元数据

每个对象单独一页，适合长期维护和版本管理。

  • 模型 / 产品 / 概念 / 场景独立存放
  • front matter 记录类型、来源、更新时间
  • 事实与观察可以分层存储
总览主树层

主树只负责“先看懂全局”

让 `overview.json` 管理一级分类与代表节点，而不是承载完整正文。

  • 稳定分类顺序
  • 控制首页信息密度
  • 让展示层具备脑图感
展示层

HTML 负责总览、专题、详情

总览页像 XMind，专题页更适合连续阅读，详情页负责精确回查。

  • 总览页：一眼看全局
  • 专题页：按主题聚焦
  • 详情页：定义 / 来源 / 判断 / 相关链接

已接入的数据层骨架

结构文件

  • `overview.json`：总览主树，前台走 `path`，底层源文件走 `sourcePath`

  • `nodes/README.md`：节点详情层说明
  • `nodes/TEMPLATE.md`：统一条目模板

建议的落地顺序

不要一上来就追求“完整图谱系统”。先把总览骨架稳定下来，再拆节点、补关系、做专题，返工最少。

1

先把 `00_AI知识总览` 立起来

把这一层定义成“总览层”，负责承接宏观地图、总览页和主树结构文件，避免继续把所有内容都堆进单个 Markdown。

2

先拆高频节点，不求一步拆完

这一轮已经先立了模板和 4 个样例节点。后续可以继续补 Claude / GPT / Agent / RAG / Cursor / WorkBuddy / ArkClaw 等高频条目。

3

最后补横向关系与专题页

当条目数量上来后，再引入更完整的 `relations.json`、专题地图页和更强的图形交互。这样展示层会越做越强，但底层结构不用推翻。
