一句话总览

如果只按术语表去背词，这一页会显得很硬。更自然的理解方式，是先顺着普通读者最容易感受到的能力变化去看：系统先是“一问一答”，再逐渐变成“能连续思考、能查资料、能调工具、能接外部系统、还能被平台封装”。在这条通俗主线之后，再回头分清基础能力层、知识增强层、执行编排层、接入协议层、产品封装层，严谨性和可读性就能同时保住。

先按视频式的理解顺序读一遍，会更容易把这些词串起来：

一问一答：先让模型“会说”

最早被感知到的能力，是你问一句，它答一句。这里最核心的是 LLM + Prompt：模型负责生成，提示词负责告诉它怎么答。

连续思考：不只一句话说完就断

当系统开始记住历史、承接上下文、跨多步继续推进任务时，背后通常就进入了 Context / Memory 的管理问题。

从会说到能查：开始补外部知识

模型光靠参数记忆不够，于是要学会去查。这里对应的是 Search / Retrieval / RAG：不是只会说，而是能结合外部资料来回答。

从能查到能做：开始调工具、走流程

再往前一步，系统不仅回答问题，还会请求外部动作执行任务。这时对应的是 Tool Use / Workflow / Agent 这一层。

能接：把能力标准化接到外部系统

当 AI 应用要稳定连接工具、资源、服务时，就会遇到 MCP 这类接入标准。它解决的是“怎么统一接”的问题。

能封装：把整套体验打包成产品能力

最后用户真正感受到的，往往不是单个术语，而是被平台封装后的整体体验，例如 Skill / OpenClaw / SubAgent 这类更贴近具体产品语境的能力包装。

严谨校准：上面这段顺序是帮助理解的认知路径，不是技术史，也不是行业统一标准层级。它解决的是“为什么这些词会一个个冒出来”这个问题；至于这些词在工程里分别属于哪一层，还要看下面的结构划分。

如果把上面的通俗主线再压缩成一条更短的理解链，可以先记成：

1

核心术语定义

LLM / Prompt / Context / Memory：属于基础能力层，但没有一篇统一“总定义”文档

  • 把大模型底层概括成“下一个 token 预测”是可以的，但只够解释生成机制，不足以完整解释今天对话系统的全部可用行为。
  • 真实可用体验通常还叠加了指令微调、偏好对齐、工具使用训练，以及具体应用的系统提示词与上下文管理。
  • Memory 也不应只理解成“把历史对话压缩后再喂回去”；它更广义地指系统跨轮保留有用状态的机制，包括摘要、偏好、任务状态和外部存储。
说明：这一组术语并没有单一官方总纲文档，本页只给出更稳妥的工作定义，不把它们包装成某种统一标准。

Search / Retrieval / RAG：RAG 有原始论文定义，不能简化成“联网搜一下”

  • Search 更偏外部来源查询；Retrieval 更偏从知识库中召回相关片段；RAG 则是让检索结果参与最终生成。
  • RAG 的核心不是“把搜到的内容贴给模型”，而是“把非参数化记忆与生成模型联动起来”。
  • 工程上通常还要考虑分块、向量化、索引、重排、来源引用、上下文窗口限制等问题。
一手来源：Lewis 等人的论文 Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks（2020）。

Tool Use / Function Calling：重点是“模型输出结构化调用意图”

  • 更稳妥的说法不是“模型直接执行了工具”，而是“模型判断何时用工具，并产出结构化调用请求”。
  • 工具的定义、执行、结果回传，通常仍由应用或平台侧来负责。
  • 不同厂商叫法略有差异：有的叫 Tool Use，有的叫 Function Calling，但本质都围绕结构化工具调用展开。
一手来源：Anthropic 官方文档 Tool use with Claude - Overview 明确区分了模型负责生成调用请求，应用负责定义与执行工具。

Workflow / Agent / SubAgent：它们属于执行编排层，不是同一种东西

  • Anthropic 的工程定义里，Workflow 是“LLM 和工具在预定义代码路径中被编排”；Agent 则是“模型动态地指导自己的过程和工具使用”。
  • 所以 Agent 不只是“把自然语言翻译成工具调用流程”，还包括任务分解、状态维护、多步循环、终止判断，以及根据环境反馈调整下一步。
  • SubAgent 可以理解为平台提供的任务委派与上下文隔离机制，但它不是跨平台统一协议，也不应该被写成“魔法增强”。
一手来源：Anthropic 官方工程文章 Building Effective Agents；Claude Code 官方文档 Create custom subagents。

MCP：是 AI 应用接入外部系统的开放标准，不只是“工具调用协议”

  • MCP 官方把它描述为连接 AI 应用与外部系统的开放标准，作用类似 AI 应用的“统一接口”。
  • 它当然覆盖工具接入，但它关注的不只是“执行工具”这一点，还包括连接资源、服务和更广泛的外部能力。
  • 因此，MCP 与 Tool Use / Function Calling 不是互相替代，而是处在不同层次，完全可以同时存在。
一手来源：MCP 官方文档 What is the Model Context Protocol (MCP)?。

Skill / OpenClaw：优先按产品语境理解，不能直接升格为通用标准

  • OpenClaw 官方把自己定义为一个自托管 gateway / personal AI assistant，用来连接聊天渠道与 AI agents。
  • OpenClaw 文档中的 Skill，是该平台内的能力封装方式，有自己的目录结构、元数据和加载机制。
  • 所以在这页里，更合理的写法是：Skill 与 OpenClaw 属于产品封装层；至于“Skill”作为跨平台通用术语的标准性，仍应视为存疑。
一手来源：OpenClaw 官方文档首页与官方 Skills 文档；官方 GitHub 仓库 README 用于交叉验证产品定位。

术语对照表

这一版把原先拥挤的表格改成卡片式阅读：每张卡只回答三个问题——它是什么、它解决什么、读的时候最容易误会什么。

LLM 基础能力层
最简定义

负责理解与生成的模型本体。

它解决什么问题

把输入转成语言输出，是整个系统的核心生成单元。

阅读提醒

“下一个 token 预测”能解释底层机制，但不足以单独解释完整产品体验。

Prompt 基础能力层
最简定义

给模型的任务说明和行为约束。

它解决什么问题

让模型知道要做什么、按什么方式输出。

阅读提醒

它是方法，不是独立能力层；提示词效果还会受到模型、上下文和工具设计影响。

Context 基础能力层
最简定义

模型当前可见的信息窗口。

它解决什么问题

承载问题、历史、资料、工具结果等执行上下文。

阅读提醒

上下文不是无限的，窗口限制会直接影响系统设计。

Memory 基础能力层
最简定义

让系统跨轮保留有用状态的机制。

它解决什么问题

支持连续对话、偏好记忆、任务状态延续。

阅读提醒

不只是“压缩历史对话”，也可能是外部存储、摘要或结构化状态。

Search 知识增强层
最简定义

从外部来源查找信息。

它解决什么问题

补足模型参数之外的最新或外部事实。

阅读提醒

Search 不等于 RAG；查到信息并不等于已经正确进入生成流程。

Retrieval 知识增强层
最简定义

从知识库或索引中召回相关内容。

它解决什么问题

把相关片段找出来，供后续生成或推理使用。

阅读提醒

它更强调召回机制本身，不必然等同于“联网搜索”。

RAG 知识增强层
最简定义

让检索结果参与最终生成的架构方式。

它解决什么问题

降低纯参数记忆的局限，增强事实性与可引用性。

阅读提醒

不要把它简化成“搜一下再贴给模型”；它通常还涉及切分、索引、重排与引用。

Tool Use / Function Calling 执行编排层
最简定义

模型生成结构化工具调用意图的机制。

它解决什么问题

让模型能请求外部系统执行动作，而不是只输出自然语言。

阅读提醒

模型通常负责“决定调用”和“产出调用请求”，真正执行一般仍由应用侧负责。

Workflow 执行编排层
最简定义

按预定义路径编排模型与工具的流程。

它解决什么问题

提高稳定性、可控性和一致性。

阅读提醒

更接近“可预测的工程流程”，不等于完全自治的 Agent。

Agent 执行编排层
最简定义

由模型动态决定过程与工具使用的执行体。

它解决什么问题

处理步骤不固定、需要反馈循环和动态规划的任务。

阅读提醒

Agent 不只是“调工具”，还涉及任务分解、状态管理、终止判断和失败恢复。

SubAgent 执行编排层
最简定义

用于任务委派和上下文隔离的子执行单元。

它解决什么问题

把高噪音、高上下文消耗的工作拆出去独立处理。

阅读提醒

这是平台相关术语，不是稳定统一的跨平台标准。

MCP 接入协议层
最简定义

AI 应用接入外部系统的开放标准。

它解决什么问题

统一连接工具、资源和服务的方式，降低集成成本。

阅读提醒

它与 Tool Use / Function Calling 分层不同，两者可以同时存在。

Skill 产品封装层
最简定义

平台内封装可复用能力的一种方式。

它解决什么问题

把提示、知识、脚本、工作流或工具使用方法打包复用。

阅读提醒

不同平台里的含义差别很大，最好始终带着具体平台语境理解。

OpenClaw 产品封装层
最简定义

一个自托管的 AI 助手 / Gateway 平台。

它解决什么问题

把聊天渠道、Agent、工具和会话连接到统一网关中。

阅读提醒

它是产品或平台，不是协议名词，也不是抽象层级本身。

怎么用这版对照表：先扫一遍层级标签建立整体地图，再按卡片里的“最简定义 → 解决问题 → 阅读提醒”顺序阅读，会比横向挤在一张表里更容易形成稳定概念。

4

常见混淆点

1

演化链 ≠ 标准层级

视频叙事适合建立直觉，但知识页不能把所有词都写成同一条严格线性链路。

2

Search ≠ RAG

搜索是取信息，RAG 是检索结果参与生成；两者相关，但不是一回事。

3

Agent ≠ 工具翻译器

Agent 的关键不只是调用工具，还包括动态规划、状态维护、循环执行和反馈修正。

4

MCP ≠ Function Calling

两者都跟“外部能力”有关，但分工不同：一个偏模型调用意图，一个偏应用接入标准。

5

Skill 不应默认当作通用标准词

不同产品里的 Skill 含义差异很大，若没有指定平台语境，就应保持谨慎。

6

OpenClaw 是产品，不是协议

它更像一个自托管 AI 助手 / 网关平台，应该放在产品封装层，而不是协议层。

5

理解这些术语的几个原则

结论 A

能直接下定义的，尽量回到一手来源

  • RAG 优先回到原始论文，而不是博客二次解释。
  • MCP 优先看官方站点与官方发布说明。
  • Tool Use / Agent / SubAgent 优先看厂商官方文档和官方工程文章。
结论 B

对平台术语，只在平台语境内下定义

  • Claude Code 的 SubAgent 有一手定义，但那是 Claude Code 语境下的定义。
  • OpenClaw 的 Skill 也有一手定义，但那是 OpenClaw 平台内的能力封装概念。
  • 平台内成立，不代表跨平台自动成立。
结论 C

没有一手来源时，宁可标“存疑”，也不要讲得像定论

  • Prompt / Context / Memory 在本页只作为工作定义使用，不强装成统一标准。
  • “Skill”作为泛化术语的跨平台通用性仍然存疑。
  • 知识库长期可维护，往往比一时说得顺更重要。

30 秒复述：

这期视频最大的价值，是把一串容易被神化的新词重新拉回工程现实；但如果要写成能长期复用的知识页，就不能把 Skill、MCP、RAG、Agent、OpenClaw 当成同一条标准演化链。更稳妥的理解是：LLM、Prompt、Context、Memory 属于基础能力层；Search、Retrieval、RAG 属于知识增强层；Tool Use、Workflow、Agent、SubAgent 属于执行编排层；MCP 属于接入外部系统的开放标准；Skill 和 OpenClaw 则更偏具体平台的产品封装。凡是有官方文档或原始论文的，就按一手来源定义；没有一手来源的，就明确标注为工作定义或通用性存疑。
