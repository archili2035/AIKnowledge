1

视频主线（按时间脉络）

从"只能一问一答"到"看起来会连续思考"

  • 大模型原生能力是"下一个 token 预测"
  • 为了让它更像助手，人们引入：
    • Prompt（提示词）
    • Context（上下文）
    • Memory（通过历史对话压缩后继续喂入）
  • 于是它从"单轮问答"变成"可追问、可延续"的系统体验

从"会说"到"能查"

  • 仅靠模型参数，信息会过时、会胡说
  • 引入搜索/检索后，模型可以使用参数外信息
  • RAG 的本质：
    • 检索相关片段 → 注入上下文 → 再生成答案

从"能查"到"能做"

  • 为了让系统执行动作（搜网页、读文档、调工具），引入 Agent
  • Agent 负责把"自然语言意图"翻译成"工具调用流程"
  • 早期很多 Agent 的"智能"并不神秘，本质是流程编排 + 提示词策略

两种"约定"被分清

Function Calling

约定"模型如何按结构化格式（如 JSON）返回调用意图"

发生在：模型 ↔ Agent

MCP

约定"Agent如何发现工具、调用工具、收参数与返回值"

发生在：Agent ↔ 工具服务

结论：两者不冲突，也不是替代关系，是不同层的协议。

Workflow / Skill / 纯 Agent 的取舍

作者给出一条"从刚到柔"的连续谱：

硬编程
Workflow
Skill
纯 Agent
✓ 最稳定、最刚性 ✗ 最灵活、最不可控

核心矛盾：灵活性 ↑，通常可控性 ↓，token 成本 ↑

SubAgent 的本质

  • SubAgent 不是魔法增强，核心价值是：
    • 任务拆分
    • 上下文隔离
    • 降低主上下文污染和膨胀

作者重点"拆穿"的误区

1

把名词当能力

新词不等于新智能。很多时候只是工程封装的演进。

2

混淆 MCP 和 Function Calling

它们作用层不同，一个在模型- Agent 之间，一个在 Agent-工具之间。

3

把 Agent 神化

很多场景只是更自动的流程路由，背后仍是规则与提示词工程。

4

低估工程成本

越"自主"的系统，越容易 token 爆炸与链路失控。

4

最有价值的洞察

洞察 A

AI 工程的本质是"把不确定和确定分层处理"

  • 确定性部分：用程序、脚本、固定流程处理
  • 不确定性部分：交给模型做语义理解、分流决策
洞察 B

真正竞争力在"体验封装"，不是"术语堆叠"

  • 普通用户要的是：开箱即用、少配置、可见可控、成本可预期
  • 名词堆叠不会带来用户价值，好的封装才会
洞察 C

短期会是"多范式并存"，不是单一范式统治

  • 稳定高频任务：偏 Workflow / Skill
  • 高不确定探索任务：偏 Agent
  • 企业落地：往往是混合架构

30 秒复述模板：

这期视频本质在讲：Skill、MCP、RAG、Agent 不是同层概念，而是一条从"会说"到"能做"的工程链路。RAG 解决"查"，Agent 解决"编排执行"，Function Calling 约束模型输出格式，MCP 约束工具调用协议。Skill 是把常见流程打包，SubAgent 是做上下文隔离。名词会变，但方法论不变：把确定性交给程序，把不确定性交给模型，并在灵活性、可控性和成本之间做平衡。
